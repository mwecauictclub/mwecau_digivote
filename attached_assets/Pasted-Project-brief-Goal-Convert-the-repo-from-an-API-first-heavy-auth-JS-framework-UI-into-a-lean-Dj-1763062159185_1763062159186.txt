Project brief — Goal

Convert the repo from an API-first, heavy-auth, JS-framework UI into a lean Django server-rendered app.
Keep a minimal JSON API only for: (1) election submission and (2) retrieving election results.
Preserve existing voter-token design and voting integrity logic.
Remove nonfunctional/slow dependencies (heavy JWT, CORS, swagger, frontend frameworks) and convert UI code to Django templates + function-based views.
Scope — in-scope

Remove/disable heavy dependencies: rest_framework_simplejwt, token_blacklist, drf_yasg, corsheaders, major frontend bundles.
Simplify REST framework config to Session/BasicAuth; keep REST framework only for the two election endpoints.
Convert UI endpoints from class-based views (CBVs) to function-based views (FBVs) that render Django templates.
Keep voter token generation/validation and voting business logic unchanged unless it blocks migration.
Update settings, requirements, and docs (.env.example).
Add tests covering voter token flow, election submission API, and results retrieval.
Scope — out-of-scope

Redesign voter-token schema or cryptographic mechanisms.
Reimplement complex external integrations (email providers) beyond minimal config checks.
Replace database engine or migration of large datasets.
Technical requirements

Python/Django server-rendered templates (Django template language) for all UI pages.
Minimal REST API endpoints:
POST /api/elections/{id}/submit/ — submit election ballot (auth via session + voter token in payload)
GET /api/elections/{id}/results/ — retrieve results (JSON)
Authentication:
UI: Django session authentication (login view, CSRF protection).
API endpoints: SessionAuthentication + additional voter-token verification where required.
Remove CorsMiddleware, JWT settings, Swagger UI config, and other unused middleware.
Preserve custom AUTH_USER_MODEL and custom authentication backend.
Keep static/media config and simple, reproducible environment variables.
Tests: use Django TestCase + APIClient for the two API endpoints.
